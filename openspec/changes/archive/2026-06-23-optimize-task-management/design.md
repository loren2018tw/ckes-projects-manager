## Context

現有任務管理系統以列表形式顯示任務，狀態由完成日期有無隱含推斷。使用者缺乏視覺化進度追蹤（看板）及人員指派能力，導致團隊協作資訊斷層。所有資料儲存於使用者 Google Drive，前端為 Quasar (Vue 3) SPA。

## Goals / Non-Goals

**Goals:**

- 任務資料模型擴充：加入開始日期、明確狀態枚舉、描述、負責人
- 看板視圖：四欄（被阻擋、未開始、進行中、已完成），任務以按鈕呈現
- 相依性自動分類：前置任務未完成 → 自動歸入「被阻擋」
- 儲存衝突判斷：狀態切換與日期邏輯一致性檢查
- 甘特圖配合新欄位更新

**Non-Goals:**

- 不引入 WebSocket 即時協作（手動重新整理即可）
- 不支援任務拖曳排序（僅拖曳變更狀態欄位）
- 不支援任務子任務（subtask）階層
- 不更改現有 OAuth 及 Google Drive API 儲存機制

## Decisions

- **狀態枚舉取代隱含推斷**：原本以完成日期有無判斷完成與否，改為明確的 `status: 'not_started' | 'in_progress' | 'completed'`。向後相容：既有無狀態任務匯入時預設為 `not_started`，完成日期不為空則設為 `completed`
- **被阻擋為動態計算欄位**：`blocked` 不儲存為獨立狀態，而是每次渲染時依前置任務完成狀態動態計算。任務實際儲存狀態僅三種，看板中的「被阻擋」為虛擬欄
- **看板與列表並存**：預設顯示看板視圖，使用者可切換回傳統列表。兩者共用同一份任務資料來源
- **拖曳變更狀態**：使用 Quasar 的 QDrag 或原生 HTML5 Drag & Drop，拖曳至不同欄位時觸發狀態更新 API
- **衝突判斷採前端即時檢查**：儲存前先驗證（如前置任務未完成時不可設為已完成），避免無效狀態組合寫入 Google Drive
- **負責人欄位存 ID**：`assignee` 儲存專案工作人員的 ID（對應至 project-staff spec 的資料），前端由工作人員清單解析顯示名稱

## Risks / Trade-offs

- [相依循環] 動態計算「被阻擋」時若存在循環相依，可能導致無限渲染 → 沿用現有循環偵測邏輯，計算時跳過已遍歷節點
- [Google Drive 延遲] 看板頻繁拖曳觸發多次寫入，可能因 Google Drive API 延遲造成狀態不一致 → 引入簡易 debounce（300ms）與樂觀更新
- [資料遷移] 既有任務無開始日期與狀態 → 提供一次性遷移邏輯，開始日期預設為截止日期前 7 天，狀態依完成日期推斷
